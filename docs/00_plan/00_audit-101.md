# Audit 101 — AI Provider & Agents SDK Inconsistencies

**Date:** 2026-06-29
**Scope:** AI model provider routing, Agents SDK usage, LLM calling patterns
**Target State:** OpenRouter (auto-free models) as primary → Cloudflare Workers AI as fallback. No Groq.

---

## 1. Provider Routing — No Multi-Provider Abstraction

### Problem
Every LLM call in the codebase hardcodes a single provider. There is no routing layer that selects a provider based on availability, cost, or configuration.

| File | Hardcoded Provider | Lines |
|------|-------------------|-------|
| `apps/api/agent/src/agents/TajiAgent.ts` | Groq (primary), Workers AI (fallback) | 293–303, 305–313 |
| `apps/api/agent/src/agents/ElimAgent.ts` | Groq (primary), Workers AI (fallback) | 39–45 |
| `apps/api/agent/src/lib/groq.ts` | Groq only | 16–48 |
| `apps/api/docgen/src/pipeline/extractor.ts` | Groq only | 28–179 |
| `apps/api/docgen/src/pipeline/converters/docx-to-markdown.ts` | Groq only | 12–44 |
| `apps/api/docgen/src/pipeline/converters/docx-to-schema.ts` | Groq only | 17–38 |
| `apps/api/docgen/src/pipeline/converters/pdf-to-schema.ts` | Groq only | 20–38 |
| `apps/api/docgen/src/pipeline/converters/vision-to-schema.ts` | Workers AI (vision) + Groq (schema) | 23, 41 |

**OpenRouter has zero references in the entire codebase.**

### Solution
Create a unified `LLMService` that:
- Accepts a provider name (`'openrouter' | 'workers-ai'`)
- Resolves the base URL, API key, and model ID per provider
- Falls through providers in priority order on failure: OpenRouter → Workers AI
- Standardizes request/response shapes
- All existing Groq code paths are deleted — not converted

## 2. Provider Type Definitions Mismatch

### Problem
Types and docs list `'groq' | 'cloudflare-ai' | 'openai'` but OpenRouter is absent and OpenAI is unimplemented.

| Location | Defined Providers | Status |
|----------|-----------------|--------|
| `packages/types/src/index.ts:75` | `'groq' \| 'cloudflare-ai' \| 'openai'` | Missing OpenRouter |
| `apps/api/agent/src/db/schema.ts:10` | Default `'groq'` | Same missing types |
| `docs/agents/agent-model.md:17` | `'groq' \| 'cloudflare_ai' \| 'openai'` | Uses underscore variant |
| `docs/architecture/platform.md:29` | `"groq" \| "cloudflare_ai" \| "openai"` | Same underscore variant |

The type file uses `'cloudflare-ai'` (hyphen) while docs use `'cloudflare_ai'` (underscore). These are inconsistent string literals that would never match each other.

### Solution
- Replace `'groq' | 'cloudflare-ai' | 'openai'` with `'openrouter' | 'workers-ai'` in `@repo/types`
- Drop Groq and OpenAI from the type union entirely
- Update all docs to use the same string values
- Fix the hyphen/underscore inconsistency: use `'workers-ai'` (hyphen) everywhere

## 3. Inconsistent LLM Calling Patterns Between Agents

### Problem
TajiAgent and ElimAgent call LLMs in different ways, despite both extending `Agent`.

| Aspect | TajiAgent | ElimAgent |
|--------|-----------|-----------|
| API Key source | `agentConfig.groqApiKey ?? env.GROQ_API_KEY` | `env.GROQ_API_KEY` only |
| Model selection | `agentConfig.modelId ?? 'llama-3.3-70b-versatile'` | Hardcoded `'llama-3.3-70b-versatile'` |
| System prompt | `agentConfig.systemPrompt ?? TAJI_SYSTEM_PROMPT` | Hardcoded `ELIM_PROMPT` constant |
| Max tokens | 512 | 768 |
| Temperature | 0.7 | 0.5 |
| Fallback | `cfAI()` method (Workers AI) | Inline Workers AI call |
| Reusable client | Custom fetch (not using `lib/groq.ts`) | Custom fetch (not using `lib/groq.ts`) |

Both agents currently use Groq-specific config and hardcoded Groq endpoints. Neither uses the shared `lib/groq.ts` utility — it's dead code.

### Solution
- Both agents use a shared `LLMService` class backed by OpenRouter API
- Config becomes provider-agnostic: `apiKey`, `model`, `systemPrompt`, `temperature`, `maxTokens`
- `agentConfig` interface drops `groqApiKey` → replaced with `openrouterApiKey` (falls back to `env.OPENROUTER_API_KEY`)
- The `LLMService` handles OpenRouter's OpenAI-compatible format natively
- Workers AI fallback stays as the second tier in the provider chain
- `lib/groq.ts` is deleted

## 4. Workers AI Binding Type Safety

### Problem
All Workers AI calls bypass TypeScript with `as any` casts.

```
apps/api/agent/src/agents/TajiAgent.ts:313
  (this.env.AI as any).run('@cf/meta/llama-3.1-8b-instruct', {...})

apps/api/agent/src/agents/ElimAgent.ts:45
  (this.env.AI as any).run('@cf/meta/llama-3.1-8b-instruct', {...})

apps/api/docgen/src/pipeline/converters/vision-to-schema.ts:23
  (env.AI as any).run('@cf/llava-hf/llava-1.5-7b-hf', {...})
```

Model IDs are hardcoded strings with no central registry.

### Solution
- Create a typed Workers AI client wrapper
- Define available models in a registry (`@cf/meta/llama-3.1-8b-instruct`, `@cf/llava-hf/llava-1.5-7b-hf`, etc.)
- Remove all `as any` casts

## 5. Agent SDK Version & Configuration Drift

### Problem
The `agents` SDK v0.0.97 is installed but:
- `tsconfig.json` previously extended `agents/tsconfig` which does not exist at that version (was removed in type-check fix)
- `hono-agents` v0.0.20 exports `agentsMiddleware` but the code was written for the older `honoAgents` API
- `agents-sdk` v0.0.30 is also a transitive dependency (deprecated warning during install)
- The `routeAgentRequest` import works but there's no explicit middleware layer for agents in the Hono app

### Solution
- Pin `agents` and `hono-agents` to specific versions known to work together
- Document which SDK API is being used (`routeAgentRequest` pattern vs middleware)
- Clean up the deprecated `agents-sdk` warning if possible

## 6. No Provider Failover or Circuit-Breaker

### Problem
The current fallback is a single try/catch: if Groq fails, try Workers AI. There is no:
- Retry with backoff for transient errors
- Circuit-breaker pattern to avoid hammering a failing provider
- Rate-limit detection and backoff
- Provider priority chain (OpenRouter → Workers AI → Groq)
- Timeout configuration per provider

### Solution
Implement a two-tier provider chain:
1. **OpenRouter** — try first (auto-free models, broad model selection)
2. **Workers AI** — fallback (zero egress cost, built into Cloudflare)
No Groq fallback. Each step should have configurable timeout, retry count, and backoff.

## 7. ConversationMachine Has No LLM Access

### Problem
The ConversationMachine (`apps/api/gateway/src/machine/machine.ts`) is a pure state machine that orchestrates conversation flow but cannot make AI decisions. It must delegate all LLM calls to the Agent DOs via HTTP fetch. This adds latency and complexity.

The machine handles `Collect` (data collection) state but the actual data extraction and validation is done by the Agent DOs separately.

### Solution
- Give the machine access to `LLMService` for lightweight AI decisions (intent classification, entity extraction)
- Keep heavy document processing in Agent DOs
- This avoids round-trips for simple classification tasks

## 8. Environment Variables & Configuration Gaps

### Problem
Only `GROQ_API_KEY` exists as an AI env var. The following are missing:

| Variable | Needed For | Status |
|----------|-----------|--------|
| `OPENROUTER_API_KEY` | OpenRouter auto-free models | Missing |
| `OPENROUTER_BASE_URL` | Custom OpenRouter endpoint | Missing |
| `AI_PROVIDER` | Global provider override | Missing |
| `AI_MODEL` | Default model for OpenRouter | Missing |

`.dev.vars.example` files reference `GROQ_API_KEY` instead of OpenRouter.

### Solution
- Replace `GROQ_API_KEY` with `OPENROUTER_API_KEY` in all `.dev.vars.example` files
- Add `OPENROUTER_BASE_URL` (defaults to `https://openrouter.ai/api/v1`)
- Add `OPENROUTER_API_KEY` to the env type interfaces
- Remove `GROQ_API_KEY` from all type definitions, wrangler.toml comments, and configs
- Add a default OpenRouter model config (`AI_MODEL` env var or DB field)
- Provider priority is implicit: OpenRouter always first, Workers AI fallback

## 9. No Provider URL Abstraction — Groq URLs Leaked Everywhere

### Problem
Groq API endpoints are hardcoded in **5 separate files** that will all need to be replaced with OpenRouter:

```
apps/api/agent/src/agents/TajiAgent.ts:33
apps/api/agent/src/agents/ElimAgent.ts:10
apps/api/agent/src/lib/groq.ts:3
apps/api/docgen/src/pipeline/extractor.ts:12
apps/api/docgen/src/pipeline/converters/docx-to-markdown.ts:10
```

Each also has its own copy of the request format (headers, body shape, error handling). Switching to OpenRouter means rewriting all 5 independently unless abstracted.

### Solution
- Delete the Groq-specific code path entirely (don't "convert" — replace)
- The `LLMService` owns the single OpenRouter endpoint (`https://openrouter.ai/api/v1/chat/completions`)
- `lib/groq.ts` is deleted; the `LLMService` handles the OpenRouter-compatible OpenAI-format API
- All callers invoke `LLMService.call()` instead of raw fetch
